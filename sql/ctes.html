CTEs (Common Table Expressions)

Readable, reusable building blocks for complex queries

What Is a CTE?

A CTE is a temporary, named result set that you define at the top of a query and then reference like a table. It only exists for the duration of that single query.

Think of it as giving a name to a chunk of logic so your main query reads like plain English instead of deeply nested spaghetti.

Real-World Analogy

Imagine you are writing a recipe. Instead of writing "take the mixture you made by combining flour, sugar, and eggs that you beat for 3 minutes" every time, you just say "take the batter." A CTE is like naming the batter - define it once at the top, then refer to it by name wherever you need it.

Why CTEs Beat Subqueries

  • Readability - the logic flows top to bottom, not inside out
  • Reusability - reference the same CTE multiple times in one query
  • Debugging - you can run each CTE independently to check results
  • Recursion - CTEs support recursive queries (subqueries do not)

Example: Finding High-Value Customers

Suppose you want to find customers who spent more than the average total order value.

The subquery approach (harder to read):

SQL SELECT customer_name, total_spent FROM ( SELECT customer_name, SUM(amount) AS total_spent FROM orders GROUP BY customer_name ) AS customer_totals WHERE total_spent > ( SELECT AVG(total_spent) FROM ( SELECT SUM(amount) AS total_spent FROM orders GROUP BY customer_name ) AS avg_calc );

The CTE approach (clean and clear):

SQL WITH customer_totals AS ( SELECT customer_name, SUM(amount) AS total_spent FROM orders GROUP BY customer_name ) SELECT customer_name, total_spent FROM customer_totals WHERE total_spent > (SELECT AVG(total_spent) FROM customer_totals);

Same result, half the code, twice the clarity. The CTE customer_totals is defined once and used twice.

The Syntax Pattern

SQL WITH cte_name AS ( -- your query here ) SELECT * FROM cte_name WHERE ...;

You can chain multiple CTEs with commas:

SQL WITH step1 AS (...), step2 AS (... FROM step1 ...) SELECT * FROM step2;

Key Takeaway

CTEs let you break complex queries into named, logical steps. They make your SQL easier to read, debug, and maintain. If you find yourself nesting subqueries more than one level deep, reach for a CTE instead.
